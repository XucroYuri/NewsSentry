# 公开站点 SEO / GEO 100 轮自动化设计

> 日期：2026-06-13
> 状态：设计稿
> 适用范围：`news-sentry.com` 公开站点、`frontend/public` 公共门户、双分支自动部署链、SEO / GEO 技术信号治理

## 1. 目标

本阶段目标是在 News Sentry 公开站点上建立一套可持续 100 轮运行的 SEO / GEO 自动化体系，优先优化公开网站本身，而不是优先做 GitHub discoverability、后台管理台或内容扩写工程。

这套自动化必须同时满足以下约束：

- 以公开站点技术信号为主验收口径。
- 允许自动修改代码、文档、规则文件并自动发布。
- 发布链固定为 `preview -> 验证通过 -> main -> production`。
- 每轮都必须做真实外部验证，不能只以 CI 通过作为完成依据。
- 必须先从开源 Skill 市场、Skill 索引站和官方文档中吸收规则，再映射到站内实现。
- 公开站相关的运行时读取链路必须朝“数据库优先”迁移，不能继续扩张“文件扫描 / frontmatter 解析即权威读取路径”的旧模式。
- 对于当前缺失、但又是长期优化必要前置条件的基础项，必须落到文档并在 README 中保留入口，防止后续被遗忘。

## 2. 核心定位

这不是一次性的“做几项 SEO 修补”，而是建立一个长期运行的公开站点技术 discoverability 优化流水线。

推荐内部定位：

> 面向 `news-sentry.com` 公开门户的 SEO / GEO 技术优化自动化系统。它持续从官方规则和开源 Skill 生态吸收适用规则，进行小步、可验证、可回滚的站点技术优化，并通过 preview 与 production 双环境外部验收保证真实生效。

本设计优先服务：

1. 搜索引擎对公开页面的抓取、索引、规范化理解。
2. AI 检索系统对公开页面、实体上下文、站点说明和结构化信号的读取与引用。
3. 后续公开增长、站点运营和规则演进的可持续治理。

本设计明确不以以下结果作为短期主目标：

- 不以即时自然流量增长作为每轮硬门槛。
- 不以大规模内容扩写作为前 30 轮重点。
- 不把管理后台、私有 API 或 GitHub 元数据作为本自动化的主优化对象。
- 不把继续加厚文件型运行时存储作为性能优化手段；文件只允许保留为投影、导出或迁移兼容层。

## 3. 现状判断

当前公开站点已具备可运行的公共门户和双分支自动部署能力，但从公开站 discoverability 角度看，仍处于“有产品入口，技术信号不完整”的阶段。

已确认的关键现状：

- `frontend/public/index.html` 目前只有基础 `title`、`viewport` 和 `theme-color`，缺少面向公开 discoverability 的完整 meta 体系。
- 仓库中暂未发现 `robots.txt`、`sitemap.xml`、`llms.txt` 等公开 discoverability 基础文件。
- 公开门户主体基于前端路由工作，后续需要显式处理页面级 title、description、canonical、结构化数据和非 hash discoverability 入口问题。
- 仓库现有 `preview` / `main` 双分支部署链可直接承接“先 preview，后 production”的自动发布约束。
- 部署验收文档中已经存在对线上公开页面、health、preview / production 状态的验证习惯，可复用于本自动化。
- 仓库总体架构已经在多份设计文档中明确了“SQLite / AsyncStore 优先、Markdown 降级为投影”的长期方向，但当前实际公开读取路径仍存在对 `drafts/*.md` 和 frontmatter 的回退依赖。
- 这意味着公开站的加载、聚合、详情读取和 discoverability 输出，仍可能被目录扫描、frontmatter 解析和文件回退拖慢，从而直接影响首屏速度、抓取稳定性和 SEO / GEO 技术信号。

## 4. 外部规则来源与优先级

本自动化必须先从外部规则来源吸收规则，再进行站内实现。外部来源分两层：

### 4.1 权威规则层

优先使用官方与标准化来源作为准绳：

- Google Search Central：SEO 基础、AI 搜索优化、canonical、robots、sitemap、结构化数据、meta/snippet 规则。
- `llms.txt` 标准：面向 LLM 推理期站点说明入口。

这些来源决定“什么是可信的基线规则”，用于约束自动化不被社区噪音带偏。

### 4.2 开源 Skill 与索引层

用于发现可落地的新检查项、新脚本模式和社区最佳实践，但不能直接替代权威规则层：

- 开源 Skill 市场与索引站。
- SEO / GEO 开源 Skill 仓库。
- 面向 agentic SEO / AI search optimization 的开源工具与技能集合。

它们的作用是：

- 发现新的技术检查点。
- 提供审计脚本思路。
- 提供 GEO / AI 可读优化视角。

它们不能直接决定：

- News Sentry 公开站应当发布哪些内容。
- 哪些页面应该自动生成。
- 哪些规则可越过 preview / production 闸门直接上线。

### 4.3 规则归一化原则

所有外部规则在进入站内实现前，必须先归一化为以下类别之一：

- `crawlability`：可抓取性
- `indexability`：可索引性
- `canonicalization`：规范化
- `structured_data`：结构化数据
- `ai_readability`：AI 可读性
- `internal_linking`：内链与信息架构
- `performance_visibility`：影响发现与呈现的性能信号
- `platform_prerequisite`：仓库外平台前置条件

只有归一化后的规则，才允许进入自动化执行层。

## 5. 100 轮执行策略

100 轮不平均发力，而是分阶段推进。

### 5.0 跨前 30 轮的基础约束：数据库优先运行时存储

在任何 SEO / GEO 外层优化之前，公开站点相关的内部读取链路必须逐步改成数据库优先。

这里的“数据库优先”指的是：

- 公开站查询路径的权威读取来源应为 `AsyncStore` / SQLite 或其后续数据库层。
- 信源运行态、公开投影索引、事件列表、聚合结果、详情所需窄字段，不应继续以扫描 `data/{target}/**/*.md` 或解析 YAML frontmatter 作为主路径。
- Markdown / 文件系统只保留三种角色：迁移兼容、人工投影、按需导出。

这里的“数据库优先”不强制要求立即删除所有文件型配置。迁移期内可接受：

- `config/sources/**`、`config/targets/**` 等静态配置仍作为作者维护源或导入源存在。
- 历史 `data/{target}/raw|evaluated|drafts|reviewed|archive` 仍作为 legacy projection 保留。

但以下做法不再允许继续扩张：

- 为了公开页面读取方便而新增新的 frontmatter 扫描逻辑。
- 把公开页权威字段继续直接绑定到 Markdown 文件解析。
- 让新的 SEO / GEO discoverability 输出依赖文件目录遍历才能生成。

这一基础约束的直接目标是：

- 提升公开站首屏和详情页读取速度。
- 降低目录膨胀和文件回退带来的尾部延迟。
- 让 `robots`、`sitemap`、canonical、结构化数据、来源页、实体页等输出有稳定的数据来源。

### 5.1 第 1-15 轮：数据库优先底座 + 基础 discoverability 补齐

目标：

- 先收缩公开站对文件型运行时读取的依赖，再补齐最基础的 discoverability 入口。
- 补齐公开站点最基础、最容易缺失的 discoverability 入口。
- 把当前“站点存在，但机器入口不完整”的状态推进到“站点有标准入口且可被稳定检查”。

优先项：

- 公开站读取链路的数据库优先梳理
- 公开信源 / 事件读取路径去文件扫描化
- `robots.txt`
- `sitemap.xml` 或 sitemap index
- `llms.txt`
- 站点级 meta 和分享标签
- 基础 canonical 策略
- 站点级结构化说明
- 缺口台账与 README 索引

### 5.2 第 16-35 轮：页面级技术规范化

目标：

- 让公开首页、feed、detail、source、analysis 等关键页面形成统一的技术 discoverability 规范。
- 让这些页面的权威数据来源继续向数据库层收敛，而不是在页面层追加新的文件回退。

优先项：

- 页面级 `title` / `description` 模板
- 页面级 canonical 规则
- route 到公开 URL 的稳定映射
- JSON-LD 结构化数据
- 面包屑和基础内链策略
- 对 hash 路由的 discoverability 补偿方案

### 5.3 第 36-65 轮：GEO / AI 可读优化

目标：

- 提高 AI 检索系统读取、理解、引用 News Sentry 公开站点内容的概率。
- 让 AI 可读输出建立在稳定数据库投影之上，而不是建立在不稳定文件解析链路之上。

优先项：

- `llms.txt` 内容增强
- 机器可读站点摘要
- 主题、来源、实体、日期等上下文信号的显式表达
- 面向引用的摘要块和可解释结构
- 减少必须依赖浏览器执行后才能理解的关键说明

### 5.4 第 66-100 轮：持续规则吸收与稳定性优化

目标：

- 自动跟进外部规则变化。
- 修复 drift、补平台前置条件、清理遗留缺口。
- 将规则更新从“人工记忆”转化为“可持续账本”。

优先项：

- 新外部 Skill 规则吸收
- 旧规则淘汰与重评分
- 线上 discoverability 漂移修复
- 性能 / 可抓取稳定性优化
- 平台级缺口清零推进

## 6. 单轮自动化执行流

每一轮自动化固定执行以下流程，不允许跳步：

1. 读取上一轮状态  
   读取自动化记忆、轮次进展账本、缺口台账和当前阶段。

2. 拉取外部规则  
   从白名单来源中抓取最新适用规则，过滤噪音与无关项。

3. 归一化与优先级排序  
   将规则归并到技术类别，并按以下优先顺序排序：  
   `可抓取性 > 可索引性 > 规范化 > 结构化数据 > AI 可读入口 > 内链与信息架构 > 性能与稳定性`

4. 生成本轮候选改动  
   只允许形成“小步、可验证、低回滚成本”的改动包。单轮原则上只推进一个主改动包。

5. 本地修改与最小必要验证  
   自动修改代码、文档、规则文件和脚本；运行构建、相关测试和静态检查。
   如果本轮涉及公开读取链路，还必须检查是否减少了文件权威读取，而不是新增了文件回退。

6. 推送 `preview`  
   自动提交并推送到 `preview` 分支，触发 preview deploy。

7. Preview 外部验收  
   必须做真实外部校验，而不是只看 workflow 绿色。

8. 自动推进 `main`  
   仅在 preview 外部验收全部通过时，自动合并 / 推送到 `main`。

9. Production 外部验收  
   在 production 重做同类公开校验，确认线上已经真实生效。

10. 更新账本与记忆  
    写回轮次进展、规则来源、缺口台账和自动化记忆。

## 7. 成功判定

本自动化默认以技术信号为主验收口径。

一轮成功必须同时满足：

- 相关构建、测试、静态检查通过。
- preview 对外可访问并通过外部技术验收。
- production 对外可访问并通过外部技术验收。
- 本轮采用的规则来源、改动内容和缺口项已经写回文档与自动化记忆。
- 没有新增但未记录的必要前置条件缺口。
- 若本轮触达公开读取链路，没有新增“文件扫描 / frontmatter 解析”作为新的权威读取依赖。

一轮失败的典型情况：

- 改动仅在本地生效，preview 或 production 未生效。
- 外部 Skill 规则未经归一化和权威规则校验就被直接落实。
- 文件已生成，但站点未实际暴露该入口。
- 发现前置条件缺失，却未写入缺口台账和 README 索引。
- 为了快修公开站问题，反而把新的公开读取逻辑继续绑到文件系统和 frontmatter 上。

## 8. Preview / Production 验收闸门

本设计将发布闸门视为自动化核心，而不是附属流程。

### 8.1 Preview 闸门

在 `preview` 上必须校验：

- `robots.txt` 可访问且内容符合当前规则。
- `sitemap` 可访问且包含应暴露的主要公开入口。
- `llms.txt` 可访问且能提供站点摘要与关键入口。
- 首页 HTML 包含预期 meta、canonical 和分享标签。
- 关键公开页面具备正确 title、description 和 canonical。
- 关键页面 JSON-LD 结构化数据存在且语法有效。
- 公开页面能正常加载，无关键 console / page error。

### 8.2 Production 闸门

在 production 上必须重复同类校验，确认 preview 成果真实进入线上。

如果 preview 通过而 production 失败，本轮仍视为失败，并必须把失败原因写入自动化记忆与进展账本。

## 12. 风险与约束

### 12.1 规则噪音风险

社区 Skill 很容易混入未经验证的“AI 搜索偏方”。处理原则是：

- 官方规则优先。
- Skill 只提供补充视角与脚本模式。
- 未归一化、不具备可验证性的规则不得直接实施。

### 12.2 过度自动化风险

允许自动改代码和自动发布，意味着闸门必须足够强。处理原则是：

- 单轮只做小步改动。
- preview 与 production 都要做真实外部校验。
- 平台级缺口不能用代码层假装解决。

### 12.3 内容越界风险

本阶段重点是技术信号，不是自动扩写大量页面。处理原则是：

- 前 30 轮尽量不将批量内容生成作为主手段。
- 对内容侧优化仅在确有结构化必要时以“小块补位”方式引入。

### 12.4 平台前置条件风险

有些问题无法仅靠仓库代码解决，例如 Search Console、Bing、Cloudflare、缓存与域级配置。处理原则是：

- 明确写入缺口台账。
- README 保留入口。
- 不因为“不能自动修”就从记录中消失。

### 12.5 存储迁移风险

公开站 SEO / GEO 会自然推动更多页面、索引和机器可读输出；如果底层仍以文件为权威读取，就会把性能和一致性问题越放越大。处理原则是：

- 公开站新增读取能力默认先找数据库落点，再决定是否保留文件兼容层。
- 静态配置文件与历史投影文件可以保留，但运行时权威读取必须逐步收敛到数据库。
- 若发现某类页面或输出仍严重依赖 frontmatter 解析，应优先进入缺口台账并在前 30 轮解决。

## 13. 测试与验证策略

验证策略分三层：

### 13.1 本地层

- 前端构建
- 与 SEO / GEO 相关的单元测试
- 规则文件校验
- 审计脚本运行
- 公开读取链路是否仍以数据库优先的回归检查

### 13.2 Preview 外部层

- `curl` / HTTP 头 / HTML 抽样
- 关键 discoverability 文件可访问性检查
- 关键页面 meta / canonical / JSON-LD 检查
- 页面加载与错误检查

### 13.3 Production 外部层

- 与 preview 相同的公开校验重复执行
- 确认真实线上状态与 preview 一致

## 14. 不做事项

本设计明确不做：

- 不把 GitHub discoverability 作为本自动化主对象。
- 不把后台管理台 discoverability 作为公开优化重点。
- 不在没有明确产品决策时自动生成大规模内容库存。
- 不以未验证的 AEO / GEO 社区说法替代官方规则。
- 不用“本地成功”替代“线上生效”。
- 不把文件型运行时存储继续当作公开站性能优化的长期基础。

## 15. 实施顺序

推荐实施顺序：

1. 落地设计文档与治理框架。
2. 把“数据库优先、文件降级为投影/导出/兼容层”纳入公开站基础约束。
3. 建立文档目录、自动化记忆和进展账本。
4. 建立最小版规则同步与技术审计脚本。
5. 建立 preview / production 技术验收脚本。
6. 先清理公开读取链路中的文件型权威依赖，再进入基础 discoverability 文件和页面级 SEO 基础设施阶段。
7. 在稳定后再逐步扩展到页面级规范化、GEO 可读优化与持续规则更新。

## 16. 成功标准

完成这一设计阶段后，应满足：

- 自动化目标、边界、闸门和文档结构已经明确，不再依赖口头约定。
- 未来 100 轮可以按统一流程持续推进，而不是每轮重新发明规则。
- 技术信号优先的验收口径已经固定，可支撑公开站长期 discoverability 优化。
- 缺口项有稳定落点，并能从 README 快速索引，避免后续被忽略。
