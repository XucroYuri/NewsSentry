## 背景

NewsSentry v2 是**批处理管道**架构：RSS 每 15 分钟（已调整为 2 分钟）轮询一次，事件顺序通过 collect → filter → judge → output 四个阶段。从新闻发布到分析师可见，延迟在分钟级。

对于需要分钟级态势感知的场景（突发事件、政策声明），需要：
1. 高价值事件入库后 30 秒内推送告警通知到浏览器
2. 事件列表/仪表盘数据自动刷新，无需手动操作

其中第二点已有 SSE 基础设施，只需缩短轮询间隔。**本 ADR 聚焦第一点：告警推送架构。**

## 决策

**采用 in-process EventBus + NotificationEngine 架构，在批处理管道的 judge 阶段边界插入 publish 钩子，而不是新建并行流式管道。**

### 架构总览

```
RSS Collector → bounded_run() (批处理管道，不变)
                    │
       judge 阶段完成后:
       event_bus.publish("news.judged.{target}", ...)
                    │
                    ▼
  ┌─────────────────────────────────┐
  │         EventBus                │
  │  in-process async pub/sub       │
  │  topic: {domain}.{stage}.{id}   │
  └──────────┬──────────────────────┘
             │ subscribe("news.judged.*")
             ▼
  ┌─────────────────────────────────┐
  │      NotificationEngine         │
  │  规则匹配 → 去重检查 → 静默时段 │
  │  → dispatch(channel, ...)       │
  └──┬──────────┬───────────────────┘
     │          │
     ▼          ▼
  browser    email (SMTP, 待实现)
  (WebSocket)
```

### 关键设计决策

#### 1. In-process EventBus，非 Redis/Kafka/NATS

**选择：** `asyncio.Queue` 实现的零外部依赖 pub/sub。

**弃用方案：**
- Celery + Redis：引入两个新依赖，单 VPS 不需要分布式任务队列
- Kafka/Redpanda：需要 2GB+ 内存，杀鸡用牛刀
- MQTT/NATS：仍增加外部依赖；In-process EventBus 零运维成本

**风险评估：** EventBus 不支持多进程。未来多 worker 时需切换到 Redis。`EventBus` 接口设计为 backend-agnostic，切换只需实现 `RedisEventBus(EventBus)`。当前单 VPS 部署不超过 10 events/min 通过 EventBus，单进程完全够用。

#### 2. 在批处理管道插入钩子，非全管道流式化

**选择：** 保持 `bounded_run()` 批处理不变，在 judge 阶段完成后调用 `event_bus.publish()`。

**弃用方案：** 全管道流式化（StreamPipeline + WriteBuffer）—— 经过三轮自审发现三个问题：
- SQLite 写入不是瓶颈（0.67 writes/s 峰值 vs 100-200 容量）；WriteBuffer 是伪需求
- 逐条 AI 调用反而增加延迟；judge 需要 mini-batch
- 四个"实时触点"中只有一个（告警推送）真正需要新架构

#### 3. 通知规则存储选 SQLite，非 YAML

**选择：** 规则存储在 `notification_rules` 表，通过 Admin API CRUD 管理。YAML 仅作初次设置的 seed。

**弃用方案：** 纯 YAML 文件配置 —— 无法支持前端 UI 动态管理、无版本历史、无并发控制。

**风险评估：** 用户数少（<50 内部分析师），规则数少（<500），SQLite 读延迟可忽略。

#### 4. WebSocket 仅用于管理后台，不推广到公开阅读器

**选择：** 管理后台用 WebSocket（双向，支持告警确认），公开阅读器继续用 SSE。

**原因：** 公开读者不需要实时，SSR + CDN 缓存是更优解。SSE 已有基础设施（`api_server.py:_sse_queues`）。

#### 5. Webhook 端点独立于管道

**选择：** Webhook 接收端点直接写入 SQLite + 发布 EventBus topic，不耦合批处理管道。

**原因：** Webhook 事件先持久化再发布。即使进程在 publish 后、消费者处理前崩溃，事件已落盘。由批处理管道的下一次 run 统一做 judge。

## 后果

### 正面

- 分析师 30 秒内收到高价值事件告警（vs 此前 15 分钟）
- 前端事件列表/仪表盘通过 SSE 自动刷新（轮询间隔从 15 分钟缩短到 2 分钟）
- 所有新增模块可独立关闭（无配置时不启动），对批处理管道零性能影响
- 推送的是"信号"，最终研判仍由人完成（Iron Man 套装定位）

### 负面

- 系统新增 6 个模块，复杂度上升
- 单进程 EventBus 未来多 worker 时需要迁移到 Redis
- 告警疲劳风险（通过 throttle_seconds + quiet_hours 缓解）

### 缓解措施

- 每规则必须有 `throttle_seconds`（默认 1800s）
- 支持 quiet hours 时段
- 浏览器通知默认只推送 `value_score >= 70` 的事件
- 结构化指标日志记录 `alert.latency_ms` 供后期评估
