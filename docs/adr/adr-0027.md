## 背景

ADR-0025 确认 FastAPI + Vanilla JS 是合法的可选 Web UI 模式，并明确不引入 React/Vue/Svelte/Angular 等重型前端框架。该决策在管理后台、CLI-first 产品阶段是合理的。

但生产站点公开门户已经出现新的产品要求：

- 站点首屏过度对齐管理宏观汇总，不像新闻阅读产品。
- 页面大多是窄屏工作台布局，没有真正利用桌面全宽空间。
- 自主 Vanilla JS/CSS 已经积累到较高维护成本。
- 新闻显示缺少读者闭环：来源、原文、摘要、推荐理由、关联讨论和详情路径没有形成统一体验。
- 参考 `https://aihot.virxact.com/` 后，下一阶段需要从“采集标题列表”转向“可消费的新闻情报流”。

因此，需要修订 ADR-0025 对公共门户的前端技术约束，但不能一次性推翻 CLI-first 核心和已有后台。

## 决策

**公共门户允许作为独立前端产品面试点引入 Vite + React + TypeScript + Tailwind + shadcn/ui。**

该决策仅适用于公开读者侧门户，不自动适用于管理后台。

### 保持不变

- News Sentry 核心仍然 CLI-first。
- FastAPI 继续提供 API 与静态托管。
- `NewsEvent` 存储契约不因前端重平台化而改变。
- 管理后台暂时保留现有 Vanilla JS SPA，避免同时重构两条产品线。
- Obsidian/Markdown/告警输出仍是正式输出路径。

### 新增允许

- 在仓库中新增公共前端 app，例如 `frontend/public/`。
- 使用 Vite 生成静态产物，并由 FastAPI 托管。
- 使用 shadcn/ui 作为公共门户基础组件体系。
- 使用 Tailwind tokens 管理公开门户的布局、密度、颜色、状态和响应式断点。
- 使用 TypeScript 定义公开门户视图模型，例如 `PublicNewsItem`。
- 公共新闻流必须支持轻量实时更新：优先采用 cursor/ETag 增量轮询、页面可见性暂停和退避策略，不把 WebSocket 或 SSE 作为第一阶段默认方案。

### 边界

- 不允许把后端领域模型迁移到前端框架中。
- 不允许为了 UI 快速实现而改变 API 权限边界。
- 不允许把管理后台重构混入公共门户试点。
- 不允许把 shadcn/ui 组件直接作为产品风格；组件库只解决可维护性，News Sentry 仍需自己的新闻编辑台设计语言。

## 方案

公共门户采用“新前端并行替换”：

1. 保留现有 `src/news_sentry/static/` 管理后台与旧公开路由。
2. 新增公共前端 app，先消费现有 API。
3. 后端补充公共展示 API 时，只新增 presentation shape，不改变 `NewsEvent` 存储契约。
4. 公共新闻 API 增加低负担增量获取能力，让前端可以在阅读过程中插入新新闻。
5. FastAPI 根据构建产物挂载新公共门户。
6. 公开路由稳定后，再删除旧 Vanilla JS 公共门户代码。

## 后果

### 正面

- 公共门户可以建立组件体系，降低继续手写 CSS/DOM 的维护压力。
- 可按 AIHOT 式新闻流重做信息组织，而不是继续在旧工作台上微调。
- TypeScript 视图模型能明确区分“读者字段”和“工程字段”。
- shadcn/ui 能提供成熟的交互基础，例如 tabs、sheet、command、skeleton、toast、dialog。

### 成本

- 引入 Node.js 构建链，CI/CD 需要新增前端 build/test。
- 部署产物与 service worker/cache 策略需要重新设计。
- CSP 需要支持新构建产物与内联样式策略。
- 需要维护一段过渡期内的新旧公共门户双轨。
- 实时更新需要额外设计缓存、轮询间隔、退避和阅读不中断交互，避免前端频繁请求放大服务器负担。

## 验收

- `#/news/feed` 或公开首页首屏直接展示新闻内容，而不是宏观仪表盘。
- 桌面 1440px 使用三栏或全宽内容台，有效内容宽度不低于 1100px。
- 移动 390x844 首屏能看到第一条新闻标题、摘要或推荐理由。
- 每条新闻有来源、原文链接、摘要、推荐理由、标签和详情路径。
- 用户停留在新闻流页面时，新采集或新增强的公开新闻能自动提示并增量进入列表，不要求整页刷新。
- 公共门户核心 UI 至少 80% 基于组件体系，而不是散落手写 DOM/CSS。
- 旧后台管理页不因公共门户重平台化发生行为回归。

## 备选方案

| 方案 | 结论 | 原因 |
|---|---|---|
| 继续 Vanilla JS 微调 | 拒绝 | 当前问题已是产品信息架构和维护体系问题，不是局部样式问题。 |
| 全站一次性 React 重写 | 拒绝 | 会把后台、权限、运维页面一起拖入高风险迁移。 |
| 只引入 Tailwind，不引入 React | 拒绝 | 可以改善样式，但不能解决组件状态、路由、类型和长期维护问题。 |
| 公共门户独立 React 试点 | 推荐 | 风险可控，同时对公开产品体验收益最大。 |
