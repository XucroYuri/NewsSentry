# ADR-0026: 三阶段客户端架构演进路线

> 日期: 2026-05-21 | 状态: Accepted | Phase 56

## 上下文

Phase 55 完成了 pywebview 桌面壳（系统托盘、PWA、SSE），Phase 56 完成了技术债清理。项目需要明确下一阶段的架构方向：是继续 Cloud VPS 单机部署，还是重新审视长远目标。

## 决策

确立三阶段客户端演进路线，暂停 Cloud VPS 单机部署，优先打磨本地完全体客户端。

### Phase 1: 本地完全体（当前 → v1.x）

- **技术栈**: pywebview + Vanilla JS SPA + Python 后端
- **目标**: 高性能全功能桌面客户端，单机闭环
- **继续打磨**: pywebiew 跨平台适配、前端交互增强、桌面体验优化
- **零外部依赖**: 本地运行，不需要云服务即可使用全部功能

### Phase 2: 跨平台高性能客户端（v2.0）

- **技术栈**: Tauri (Rust 核心 + Web 前端)
- **支持**: macOS / Windows / Linux 三平台原生应用
- **架构**: API-first，客户端通过 REST 与本地/云端 API Server 通信
- **CLI 定位**: news-sentry 作为本地 agent 运行，GUI 客户端管理多节点

### Phase 3: 云端集群 + 分布式采集（远期）

- **高频上云**: RSS/API 定时轮询集中部署到云端
- **低频本地**: 社媒 KOL / 自定义信源 / 个性化研判留本地
- **分布式预留**: 本地客户端可选加入分布式采集网络
- **技术选型**: TBD (libp2p / IPFS / 自研轻量协议)，Phase 3 启动时决策
- **数据主权**: 用户核心数据（研判规则、自定义信源、反馈）始终在本地

## 取舍

- ✅ 采用：pywebview 继续打磨到 Phase 1 完全体
- ✅ 采用：Tauri 作为 v2.0 目标
- ✅ 采用：分布式协议先标记，Phase 3 再决策
- ❌ 暂停：Cloud VPS 单机部署（Phase 15），直接推向云端集群模式
- ❌ 暂缓：分布式网络技术选型

## 影响

- 开发优先级从"部署上线"转向"本地客户端体验完善"
- Phase 57+ 应聚焦 pywebview 桌面壳打磨、前端交互优化、性能提升
- 架构设计保留 API-first 原则，为 Phase 2/3 预留接口兼容性
- Cloudflare Pages 前端部署继续维护（作为 Web 访问入口）
