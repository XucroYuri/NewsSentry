# Phase 49.5: 应用产品化 — 设计文档

> 状态: Draft | 日期: 2026-05-18
> 路线: 在 Phase 50 (本地客户端) 之前，将应用打磨为可日常使用的产品
> 四层结构: 数据可靠性 → 视觉翻新 → 功能补齐 → 交互打磨

---

## §1. 问题诊断

### 当前状态（v1.5.0 前端能力打通后）

News Sentry 具备完整 pipeline（163 sources、52 API endpoints、23 前端 Tab），但存在"能跑但不合用"的断层：

| 维度 | 当前状态 | 问题 |
|------|---------|------|
| 数据 | Cloudflare 容器 `total_events_all_targets: 0` | 新用户打开后看到空壳 |
| 视觉 | 功能性暗色 UI | 停留在开发者工具水平，非产品体验 |
| 功能 | 核心 CRUD 完备 | 缺用户管理、通知配置、简报发送 |
| 体验 | 基础交互 | 无引导、无骨架屏、空状态无提示 |

### 目标

将 News Sentry 从"开发者可用的工具"升级为"普通用户可日常使用的产品"：

1. **登录即看到数据** — 自动采集可靠运行 + 诊断可见
2. **前端像专业产品** — 品牌化视觉 + 信息层级清晰 + 响应式
3. **能力完备** — 用户管理 + 通知配置 + 简报发送
4. **操作无摩擦** — 引导式空状态 + 骨架加载 + 键盘操作

### §2.1 双路径部署架构

> **执行状态**：Path A 是当前实施方向（Phase 49.5 + Phase 50）；Path B 是架构前瞻，不在本阶段实施但在设计上保留演进可能。

News Sentry 的长期产品路线存在两条合理的分叉路径。本设计的职责是：**优先实现 Path A，同时在关键接入点上留出 Path B 的扩展空间**，不在今天的设计决策中锁死未来的可能性。

#### Path A: 重本地轻云端 ← 当前实施方向

本地一体化客户端承担全部采集/过滤/研判/输出工作，云端仅提供 AI API 调用。

```
┌──────────────────────────────────────────────────┐
│              用户本地设备                           │
│  ┌────────────────────────────────────────────┐   │
│  │  news-sentry serve (Phase 50)               │   │
│  │  FastAPI + SQLite + Auto Collector          │   │
│  │  localhost:PORT → 完整 Web UI               │   │
│  └────────────────────────────────────────────┘   │
│                     │                              │
│                     ▼                              │
│  ┌────────────────────────────────────────────┐   │
│  │  本地数据存储 (~/.news-sentry/data/)          │   │
│  │  用户数据完全在本机，隐私可控                  │   │
│  └────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────┘
       │
       │ AI API 调用 (ANTHROPIC_API_KEY)
       ▼
┌──────────────────────────────────────────────────┐
│  云端 AI 服务 (Anthropic / OpenAI)                 │
│  · 会员计划: 按 token 消费计费                     │
│  · API Key 作为会员凭证                           │
└──────────────────────────────────────────────────┘
```

- 用户拥有自己的硬件（NAS/桌面/小型服务器）
- 核心 pipeline 在本地 24/7 运行
- 云端仅提供 AI API 调用（会员 token 配额）
- 数据主权在用户手中
- 适合隐私敏感场景（记者、研究者、律师）
- **当前局限**：本地硬件资源决定了采集规模上限（单机处理 N 个 target × M 个信源）

#### Path B: 轻本地重云端 ← 未来演进方向

当本地一体机的采集能力达到硬件瓶颈后，将采集/过滤/研判分解到云端集群，客户端退化为轻量信息接收端。这一演进路径不削弱能力反而增强——云端大规模集群能采集远超单机的信源数量，同时客户端变得极轻（手机、IoT 设备均可接入）。

```
┌──────────────────────────────────────────────────┐
│  云端集群 (开发者运营)                               │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐          │
│  │ Collector │ │  Judge   │ │  Output  │  ...     │
│  │ (大规模)   │ │ (集群)   │ │ (分发)    │          │
│  └──────────┘ └──────────┘ └──────────┘          │
│  分解 pipeline 到多节点并行处理                     │
│  api.news-sentry.com                             │
└──────────────────────────────────────────────────┘
       │
       │ HTTPS (REST API / SSE / WebSocket)
       ▼
┌──────────────────────────────────────────────┐
│  用户端 (极轻量)                                │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐      │
│  │ Web SPA  │ │ 移动 PWA  │ │ IoT/信息屏 │     │
│  └──────────┘ └──────────┘ └──────────┘      │
│  零本地采集 · 低能耗 · 实时推送                 │
└──────────────────────────────────────────────┘
```

- 采集能力从单机 N×M 信源升级为集群级横向扩展
- 客户端从"跑采集服务"退化为"收信息"，大幅降低硬件门槛
- 手机、平板、智能音箱、车载屏幕均可成为信息终端
- 会员费包含云端集群服务 + AI token
- **不在本阶段实施**，但架构上保留 pipeline 分解→集群化的演进空间

#### 本设计对双路径的支撑策略

| 维度 | Path A (当前实施) | Path B 预留 (不锁死) |
|------|----------|------|
| API 层 | 统一 REST API | 前后端分离架构天然支持部署迁移 |
| 前端 SPA | 连接 localhost | 登录页支持远程 URL（已内置） |
| 通知渠道 | 本地直发 SMTP/webhook | 共用配置 schema，云端代理发送 |
| 离线语义 | 服务未启动引导 | 网络中断缓存 + 自动重连 |
| 数据存储 | 本地 SQLite | `NEWSSENTRY_DATA_DIR` 统一抽象 |
| Pipeline | 本地一体执行 | 采集/过滤/研判/输出可按阶段分解到独立节点 |

**不在本阶段实施的**（留给 Phase 51+）：
- Pipeline 阶段分解 + 集群调度（Path B 的核心——从一体化 runtime 拆分为可独立部署的阶段节点）
- 多租户云端服务平台（需用户隔离、配额管理、计费系统）
- PWA 移动端包装（需 Service Worker + 推送通知）
- 本地客户端系统托盘集成（P50 提供 serve 命令，托盘交给 P51+）

## §3. Layer 1: 数据可靠性

### §3.1 自动采集修复

**根因**：`api_server.py:1054` 的 `_auto_collect_loop` 调用 `bounded_run_async(target_id=target_id, stage="collect")`，仅采集单个 target 且不做 filter/judge/output，导致事件虽被采集但未完成全流程索引。

**修复方案**（与 Phase 50 Task 50.3 重叠，此处是对同一改动的引用）：

> 详见 `docs/specs/2026-05-18-phase-50-local-client-design.md` §8 Task 50.3a-50.3d。
>
> 在 Phase 49.5a 中优先完成此改动：
> - `_auto_collector_state["target_ids"]` 替换 `target_id`，读 `NEWSSENTRY_TARGET_ID`
> - `_auto_collect_loop` 调用 `bounded_run_multi_async(targets=target_ids, stage=stage)`
> - `stage` 从 `NEWSSENTRY_COLLECT_STAGE` 读取，默认 `collect`
> - `collector/status` 端点返回 `target_ids` + `stage`

### §3.2 采集诊断端点

**新增端点**：`GET /api/v1/collector/diagnostics`

```python
@app.get("/api/v1/collector/diagnostics")
async def collector_diagnostics() -> dict[str, Any]:
    """返回采集系统诊断信息，帮助排查"无数据"问题。"""
    checks = []

    # 1. 自动采集状态
    checks.append({
        "name": "auto_collect_enabled",
        "ok": _auto_collector_state["enabled"],
        "message": "已启用" if _auto_collector_state["enabled"] else "未启用 — 设置 NEWSSENTRY_AUTO_COLLECT=1"
    })

    # 2. API Key 配置
    has_ai_key = bool(os.environ.get("ANTHROPIC_API_KEY") or os.environ.get("OPENAI_API_KEY"))
    checks.append({
        "name": "ai_api_key",
        "ok": has_ai_key,
        "message": "已配置" if has_ai_key else "未配置 AI API Key — 研判/翻译将跳过"
    })

    # 3. 数据目录
    data_exists = _data_dir.exists()
    target_dirs = sorted([d.name for d in _data_dir.iterdir() if d.is_dir()]) if data_exists else []
    checks.append({
        "name": "data_directory",
        "ok": data_exists and len(target_dirs) > 0,
        "message": f"数据目录: {_data_dir} — {len(target_dirs)} 个 target: {', '.join(target_dirs) if target_dirs else '无'}"
    })

    # 4. 信源健康概览
    healthy = 0
    unhealthy = 0
    if data_exists:
        for tid in target_dirs:
            health_file = _data_dir / tid / "source_health.json"
            if health_file.exists():
                try:
                    health_data = json.loads(health_file.read_text())
                    for h in (health_data if isinstance(health_data, list) else []):
                        if h.get("healthy"):
                            healthy += 1
                        else:
                            unhealthy += 1
                except Exception:
                    pass
    checks.append({
        "name": "source_health",
        "ok": (healthy + unhealthy) > 0,
        "message": f"健康: {healthy}, 异常: {unhealthy}" if (healthy + unhealthy) > 0 else "暂无信源健康数据 — 运行一次采集后生成"
    })

    # 5. 最近采集时间
    last_run = _auto_collector_state["last_run_at"]
    checks.append({
        "name": "last_collection",
        "ok": last_run is not None,
        "message": f"最后采集: {last_run}" if last_run else "尚未执行采集 — 等待首次采集周期"
    })

    overall = all(c["ok"] for c in checks)
    return {"overall": "healthy" if overall else "attention_needed", "checks": checks}
```

### §5.2 通知渠道配置

**现状**：`alert_pipeline.py` 已支持 `generic_webhook`、`feishu`、`email` 三种推送渠道，但无 UI 配置入口。

**双路径差异**：

| 渠道 | Path A (本地) | Path B (云端) |
|------|-------------|-------------|
| email | 本地直接 SMTP 发送 | 云端 SMTP 发送（用户配置的服务器） |
| feishu | 本地直接 POST webhook | 云端 POST webhook |
| generic_webhook | 本地直接 POST | 云端 POST |
| push (未来) | N/A | 云端 push notification → PWA/mobile |

两种路径共用同一套配置 schema 和 UI，后端根据部署环境自动选择发送路径。

**后端新增端点**：

| 端点 | 方法 | 权限 | 功能 |
|------|------|------|------|
| `/api/v1/settings/notifications` | GET | admin | 获取通知渠道配置 |
| `/api/v1/settings/notifications` | PUT | admin | 更新通知渠道配置 |

**配置 Schema**：

```json
{
  "channels": {
    "email": {
```

### §6.2 Toast 通知优化

当前 Toast 2 秒自动消失，改为：

- **持久化 Toast**：操作结果（保存成功/失败）显示 5 秒
- **可关闭 Toast**：添加 ✕ 按钮
- **Toast 队列**：多个 Toast 堆叠而非覆盖
- **颜色区分**：success (绿) / error (红) / warning (黄) / info (蓝)

### §6.3 错误重试 + 离线降级

在 `api.js` 的 `api()` 函数中增加：

- **自动重试**：网络错误自动重试 2 次（间隔 1s/3s 指数退避）
- **离线检测**：`navigator.onLine` 监听，离线时显示横幅
- **离线队列**：写操作暂存到 localStorage 的 `ns_offline_queue`，恢复在线后自动重放

**两种离线场景的差异化处理**（对应 §2.1 双路径）：

| 场景 | 检测方式 | 横幅文案 | 恢复动作 |
|------|---------|---------|---------|
| Path A: 本地服务未启动 | `GET /health` 连续失败 + `navigator.onLine === true` | "本地服务未运行 — `news-sentry serve` 启动" | 轮询 `/health`，上线后自动恢复 |
| Path B: 网络中断 | `navigator.onLine === false` | "网络已断开 — 恢复连接后自动重连" | 监听 `online` 事件，触发自动重连 |

**离线降级策略**：
- Path A 离线：读操作返回缓存（如有），写操作入离线队列
- Path B 离线：读操作显示缓存数据 + "离线数据"标记，写操作入离线队列
- 两种路径共享同一个离线队列机制（localStorage `ns_offline_queue`）

### §6.4 键盘导航增强

- Tab 键在表单内循环聚焦
- 事件列表支持 j/k（上下移动）和 Enter（打开详情）
- 分页支持 ← → 翻页
- 弹窗 Esc 关闭，Enter 确认
