## 1. 研究概述

本文档对当前开源舆情监控与新闻聚合领域的三个核心参考项目进行了深度研究，旨在为 News Sentry（意大利突发新闻监控系统）的架构设计与技术选型提供依据。三个项目在舆情监控链条上覆盖不同层次：

| 项目 | 定位 | Stars | 语言 | 在我们系统中的角色 |
|------|------|-------|------|-------------------|
| **TrendRadar** (sansan0) | 轻情雷达 — 轻情监控与热点聚合 | 57,064 | Python | **参考基线**：多平台聚合 + AI分析简报 + 多渠道推送 |
| **BettaFish/微舆** (666ghj) | 多Agent舆情分析助手 | 40,822 | Python | **深度分析参考**：多智能体协作 + 情感分析 + 报告生成 |
| **OpenCLI** (jackwener) | 网站CLI化工具 | 19,380 | JavaScript | **采集增强**：确定性CLI适配器 + 登录态复用 + Agent技能 |

## 2. TrendRadar — 轻情雷达

### 2.1 项目概况

- **仓库**: github.com/sansan0/TrendRadar
- **Stars**: 57,064 | **Forks**: 24,162
- **当前版本**: v6.6.2 (MCP v4.0.2)
- **创建时间**: 2025-04-28
- **定位**: 轻量级热点新闻聚合与AI分析工具，30秒网页部署，1分钟手机通知

### 2.2 核心架构

TrendRadar 采用模块化的Python架构，核心数据流为：

```
Scheduler(时间线调度)
  → Crawler(多源抓取: 热榜平台 + RSS)
    → StorageManager(SQLite持久化, 支持S3远程)
      → AIAnalyzer(5板块分析流水线)
        → AIFilter(两阶段标签提取与分类)
          → AITranslator(多语言翻译)
            → ReportGenerator(HTML/TXT报告)
              → NotificationDispatcher(9渠道推送)
```

**关键设计模式**：
- **AppContext 外观模式**: 所有模块通过 `AppContext` 类（trendradar/context.py）访问配置、存储和工具，避免模块间直接耦合
- **双源抓取架构**: 热榜平台（via NewsNow API）+ RSS订阅源，统一数据模型
- **SQLite + S3 双模式存储**: 本地SQLite默认，支持S3兼容远程存储（MinIO/阿里云OSS等）
- **MCP Server集成**: 通过Model Context Protocol暴露完整的新闻聚合、分析和通知能力

### 2.3 数据采集层

**热榜平台抓取器**:
- 基于 [NewsNow API](https://github.com/ourongxing/newsnow) 聚合器
- 内置11个平台ID: `toutiao`, `baidu`, `wallstreetcn-hot`, `thepaper`, `bilibili-hot-search` 等
- 社区Issue #95扩展了40+平台配置（含财经、社交媒体、国际源等分类）
- 每个平台通过唯一`id`字符串标识，映射到NewsNow端点
- 请求间隔: 默认100ms + 随机抖动10-20ms，指数退避重试

**RSS抓取器**:
- 支持自定义RSS Feed配置
- 与热榜抓取结果统一合并处理

**数据模型**:
- `NewsData` / `RSSData` 容器数据类: 抓取日期时间、按源分组的条目、`id_to_name`映射、`failed_ids`错误跟踪

### 2.4 AI分析引擎

5板块分析流水线:
1. 热点趋势分析
2. 关键词热度追踪
3. 跨平台关联发现
4. 潜在影响评估
5. AI简报生成

**智能兴趣过滤**: 两阶段处理
- 阶段1: 标签提取（基于配置的兴趣关键词匹配）
- 阶段2: AI分类（判断新闻是否属于用户关注领域）

**多语言翻译**: 批量翻译机制，支持AI驱动的多语言新闻摘要

### 2.5 通知与推送

9大渠道: 企业微信、个人微信、飞书、钉钉、Telegram、邮件、ntfy、Bark、Slack

**消息分片**: 各渠道有字节限制，`split_content()` 自动按阈值分批发送

### 2.6 MCP Server架构

三层内部架构:
- **数据服务层** (DataService): 查询SQLite数据库
- **解析服务层** (ParserService): 格式化数据
- **缓存服务层** (CacheService): 请求缓存

暴露的资源:
- `config://platforms` — 所有配置的平台注册表
- `config://rss-feeds` — RSS源注册表

### 2.7 对News Sentry的适配价值

| TrendRadar能力 | 适配到News Sentry | 改造量 |
|---------------|------------------|--------|
| 多平台热榜聚合 | 替换为意大利媒体源（ANSA/RSS等） | 中 — 需要意大利特化的平台配置 |
| AI分析简报 | 直接复用，增加涉华相关性判断 | 低 |
| 多渠道推送 | 直接复用飞书/企微推送 | 低 |
| RSS抓取 | 直接复用，配置意大利媒体RSS | 低 |
| SQLite存储 | 可用，但需扩展为NewsEvent schema | 中 |
| MCP Server | 可作为AI Agent的上下文源 | 低 |

**关键局限**:
- 主要面向中国国内平台，无意大利/欧洲新闻源预配置
- 热榜抓取依赖NewsNow API（中国平台聚合器），无法直接用于国际源
- 无涉华舆情判断、多语言翻译是在分析层而非采集层
- 单一pipeline，无collect→filter→judge→output四环节拆分

## 3. BettaFish/微舆 — 多Agent舆情分析

### 3.1 项目概况

- **仓库**: github.com/666ghj/BettaFish
- **Stars**: 40,822 | **Forks**: 7,542
- **创建时间**: 2024-07-01
- **定位**: 从零实现的多智能体舆情分析系统，"人人可用的多Agent舆情分析助手"

### 3.2 核心架构 — 五引擎异构多智能体

```
                    ┌─────────────────────────────────────────┐
                    │           ForumEngine                   │
                    │    (智能主持人/协调器)                    │
                    │    监控Agent日志 → 引导辩论 → 合成观点    │
                    └───┬──────────┬──────────┬──────────────┘
                        │          │          │
              ┌─────────▼──┐  ┌───▼────────┐  ┌▼──────────────┐
              │ InsightEngine│  │ MediaEngine│  │ QueryEngine   │
              │ (数据库挖掘) │  │ (多模态理解)│  │ (网页搜索)    │
              │ +情感分析    │  │ +视频/图片  │  │ +国内外检索   │
              └──────┬──────┘  └─────┬──────┘  └──────┬───────┘
                     │               │                │
              ┌──────▼───────────────▼────────────────▼───────┐
              │              ReportEngine                       │
              │    (多格式专业报告生成: HTML/PDF/Markdown)       │
              └──────────────────────┬────────────────────────┘
                                     │
              ┌──────────────────────▼────────────────────────┐
              │              MindSpider                         │
              │    (AI驱动分布式爬虫系统)                        │
              │    BroadTopicExtraction → DeepSentimentCrawling │
              └─────────────────────────────────────────────────┘
```

### 3.3 MindSpider — 数据采集层

**两阶段架构**:

**阶段1: BroadTopicExtraction（广度话题提取）**
- 从新闻来源（微博、知乎等）自动提取热门话题
- 为每个话题生成搜索关键词
- 支持多语言关键词集、加权关键词、分层关键词分类

**阶段2: DeepSentimentCrawling（深度情感爬取）**
- 基于提取的关键词，在7大社交媒体平台深度爬取
- 支持平台: 小红书(xhs)、抖音(dy)、微博(wb)、B站(bilibili)、快手(kuaishou)、贴吧(tieba)、知乎(zhihu)
- 构建在 [MediaCrawler](https://github.com/NanmiCoder/MediaCrawler) 框架的包装器之上
- 智能关键词分配: 根据平台特性自动分配和优化关键词

### 3.4 多智能体协作机制 — ForumEngine

ForumEngine 是BettaFish最独特的设计:

1. **观察-协调模式**: ForumEngine不直接参与分析，而是监控三个Engine的日志输出
2. **观点提取**: 从各Engine的分析结果中提取有意义的贡献
3. **智能引导**: LLM主持人（Qwen3-235B）分析各Agent观点，生成协作建议
4. **辩论机制**: 引导Agent之间辩论，避免思维同质化
5. **迭代深化**: 基于协作建议，各Engine深化研究

**协作流程**:
```
Flask接收用户需求
  → 并行启动: InsightEngine + MediaEngine + QueryEngine
  → 各Engine独立分析
  → ForumEngine收集观点
  → LLM主持人生成引导
  → 各Engine基于建议深化研究
  → 迭代直至收敛
  → ReportEngine生成报告
```

### 3.5 四大分析引擎

| 引擎 | 数据源 | 关键能力 | 搜索工具 |
|------|--------|---------|---------|
| **InsightEngine** | 私有数据库 | 数据库挖掘、情感分析、聚类搜索、关键词优化 | sentiment_analyzer, keyword_optimizer |
| **MediaEngine** | 多模态内容 | 视频/图片深度理解、结构化信息卡片提取 | 多模态搜索 |
| **QueryEngine** | 网页 | 国内外新闻搜索、多语言信息检索 | 网页搜索 |
| **ReportEngine** | 聚合数据 | 多格式专业报告、模板选择、章节生成、图表渲染 | 报告模板系统 |

**节点化处理Pipeline**: 每个Engine内部采用node-based处理管线:
- `base_node` → `search_node` → `summary_node` → `report_structure_node` → `formatting_node`

### 3.6 情感分析模型

BettaFish包含完整的微调模型栈:
- **WeiboSentiment_Finetuned**: BERT-LoRA, GPT2-AdapterTuning, GPT2-LoRA
- **WeiboSentiment_MachineLearning**: Bayes, BERT, LSTM, SVM, XGBoost对比
- **WeiboMultilingualSentiment**: 多语言情感分析
- **BertTopicDetection_Finetuned**: 主题检测微调
- **WeiboSentiment_SmallQwen**: Qwen3嵌入 + LoRA微调

### 3.7 报告模板系统

6类专业报告模板:
1. 日常或定期舆情监测报告
2. 突发事件与危机公关舆情报告
3. 社会公共热点事件分析报告
4. 市场竞争格局舆情分析报告
5. 企业品牌声誉分析报告
6. 特定政策或行业动态舆情分析报告

### 3.8 数据库设计

- **技术栈**: PostgreSQL/MySQL (元数据) + 大数据存储
- **双模式设计**: 元数据管理(SA模型) + 大数据存储(BigData模型)分离
- **核心表**: 话题表、爬取任务表、平台内容表、评论数据表
- **任务关联**: `task_topic_platform` 复合索引，跟踪每个话题在每个平台的爬取状态

### 3.9 对News Sentry的适配价值

| BettaFish能力 | 适配到News Sentry | 改造量 |
|--------------|------------------|--------|
| ForumEngine多Agent协作 | 参考设计: judge环节可用类似辩论机制提升研判质量 | 高 — 需大幅简化 |
| InsightEngine情感分析 | 参考sentiment_analyzer设计 | 中 — 需适配意大利语 |
| MindSpider爬虫框架 | 不直接适用(面向中国社媒)，但两阶段架构可借鉴 | 高 |
| 报告模板系统 | 直接参考"突发事件"和"政策动态"模板 | 低 |
| 微调情感模型 | 需重新训练意大利语模型 | 极高 |
| QueryEngine多语言搜索 | 可参考其多语言检索设计 | 中 |
| Database Schema | 参考其话题-任务-平台关联设计 | 中 |

**关键局限**:
- 完全面向中国社交媒体生态（微博/小红书/抖音/B站等）
- 情感模型针对中文训练，无法直接用于意大利语
- MindSpider的MediaCrawler依赖不支持国际平台
- 系统较重（5引擎+微调模型+PostgreSQL），对轻量级监控场景过重
- ForumEngine的Flask+LLM主持人架构对于heartbeat式周期监控不太适合

## 4. OpenCLI — 网站CLI化工具

### 4.1 项目概况

- **仓库**: github.com/jackwener/OpenCLI
- **Stars**: 19,380 | **Forks**: 1,907
- **创建时间**: 2026-03-14
- **定位**: 将任意网站、浏览器会话、Electron应用和本地工具转化为确定性CLI接口

### 4.2 核心设计理念 — "编译期智能 vs 运行期智能"

这是OpenCLI最重要的设计哲学:

- **问题**: AI Agent操控浏览器时让LLM实时理解DOM/截图再决策，每次执行消耗大量Token，结果不稳定
- **解决**: 利用AI一次性为某个网站生成确定性CLI Adapter。之后每次执行该Adapter无需调用LLM
- **效果**: 成本为零（Zero LLM cost at runtime），结果确定（Same command, same output, every time）

### 4.3 三种自动化面

1. **内置适配器** — 90+预置命令（bilibili, twitter, reddit, hackernews等）
2. **AI Agent浏览器操控** — 通过`opencli-adapter-author`技能，Agent可导航、点击、输入、提取任意网站
3. **新适配器编写** — AI引导端到端：网站探查 → API发现 → 字段解码 → 代码生成 → 验证

### 4.4 浏览器桥接

- **Chrome Extension**: 直接复用用户已登录的Chrome会话
- **Daemon**: 本地守护进程，自动启动，管理浏览器连接
- **安全**: 凭证永不离开浏览器，API密钥不暴露给外部系统

### 4.6 AI Agent技能系统

5个核心技能:
| 技能 | 用途 | 对News Sentry的价值 |
|------|------|-------------------|
| `opencli-adapter-author` | 操作网站 + 编写新适配器 | 核心 — 为意大利新闻网站生成CLI适配器 |
| `opencli-autofix` | 修复破损适配器 | 维护 — 网站改版后自动修复 |
| `opencli-browser` | 浏览器自动化参考 | 基础 — Agent操控浏览器的原语 |
| `opencli-usage` | 命令与站点快速参考 | 参考 — 查询已有能力 |
| `smart-search` | 跨现有能力搜索 | 发现 — 查找已有适配器 |

### 4.7 适配器开发流程

```
1. 探查网站 → 分类模式(SPA/SSR/JSONP/Token/Streaming)
2. 发现端点 → 网络拦截/初始状态/Bundle搜索/Token追踪
3. 确定认证策略 → PUBLIC/COOKIE/HEADER/INTERCEPT
4. 解码响应字段 → 设计输出列
5. opencli browser init {site}/{name} → 编写适配器
6. opencli browser verify {site}/{name} → 验证
7. 持久化到 ~/.opencli/sites/{site}/ → 下次同站更快
```

### 4.8 对News Sentry的适配价值

| OpenCLI能力 | 适配到News Sentry | 改造量 |
|------------|------------------|--------|
| Twitter适配器 | `opencli twitter search --lang it` 搜索意大利语推文 | 低 — 直接使用 |
| Reddit适配器 | `opencli reddit search --subreddit italy` | 低 — 直接使用 |
| Google News适配器 | `opencli google news --region it` | 低 — 直接使用 |
| 新适配器开发 | 为ANSA/Corriere/Repubblica生成CLI适配器 | 中 — 需AI辅助一次性生成 |
| 浏览器登录态 | 复用已登录Chrome会话爬取社媒 | 无 — 直接使用 |
| Adapter自动修复 | 网站改版后autofix | 低 — 安装技能即可 |

**关键局限**:
- 预置适配器以中国/英文平台为主，无意大利媒体预配置
- 需要Chrome浏览器始终运行并登录目标平台
- Node.js生态，与Python为主的TrendRadar/BettaFish技术栈不同
- 适配器生成需要AI（LLM成本），但只需一次
- 社媒平台API限制（Twitter/X的rate limit等）

## 5. 三个项目的互补关系

### 5.1 能力矩阵对比

| 能力维度 | TrendRadar | BettaFish | OpenCLI |
|---------|-----------|-----------|---------|
| **数据采集广度** | 35+平台热榜+RSS | 7大社媒深度爬取 | 90+网站CLI化 |
| **采集确定性** | 低(API依赖) | 低(爬虫框架) | **高(编译期确定)** |
| **情感分析深度** | 浅(AI摘要) | **深(微调模型+多Agent)** | 无 |
| **多Agent协作** | 无 | **ForumEngine辩论机制** | 无(但可被Agent调用) |
| **涉华相关性** | 无 | 无(但有中文舆情能力) | 无 |
| **多语言支持** | 翻译层 | 多语言情感模型 | 浏览器原生多语言 |
| **报告生成** | HTML/TXT | **6类专业模板+PDF** | 无 |
| **推送渠道** | **9大渠道** | Streamlit UI | 无 |
| **部署复杂度** | 低(Docker一键) | 高(5引擎+DB+微调模型) | 中(Chrome+Daemon+Extension) |
| **LLM运行成本** | 中(AI分析) | 高(多Agent+LLM主持) | **零(编译期消耗)** |

### 5.2 集成架构构想

三个项目在News Sentry中可以形成互补的分层架构:

```
┌──────────────────────────────────────────────────────────────────┐
│                    News Sentry 编排层                             │
│  heartbeat engine + pipeline builder (collect→filter→judge→output)│
└──────────┬───────────────┬────────────────┬─────────────────────┘
           │               │                │
  ┌────────▼────────┐ ┌───▼──────────┐ ┌───▼───────────────┐
  │   OpenCLI层      │ │ TrendRadar层 │ │  BettaFish层       │
  │  (确定性采集)    │ │ (聚合+推送)  │ │  (深度分析)        │
  │                  │ │              │ │                    │
  │ • 意大利新闻网站 │ │ • RSS聚合    │ │ • 情感分析(需适配)  │
  │ • 社媒CLI适配器  │ │ • AI简报     │ │ • 多Agent研判      │
  │ • 登录态爬取     │ │ • 多渠道推送 │ │ • 专业报告生成      │
  │ • Adapter自修复  │ │ • 关键词筛选 │ │ • 报告模板         │
  └──────────────────┘ └──────────────┘ └────────────────────┘
```

## 6. 对News Sentry架构设计的具体建议

### 6.1 采集层: OpenCLI + RSS + World News API

**推荐方案**: 三源互补采集

| 采集源 | 方式 | 覆盖范围 | 实现路径 |
|--------|------|---------|---------|
| **意大利新闻网站** | OpenCLI适配器 | ANSA, Corriere, Repubblica, Il Sole 24 Ore | `opencli-adapter-author`一次性生成 |
| **社媒平台** | OpenCLI预置适配器 | Twitter/X(意语), Reddit(r/italy), Facebook(意语群组) | 直接使用预置命令 |
| **国际组织** | OpenCLI适配器 + RSS | FAO, WFP, UNESCO等驻意组织 | 优先RSS，无RSS则OpenCLI |
| **通用新闻API** | World News API | 292个意大利新闻源 | API调用，作为基线源 |
| **RSS订阅** | TrendRadar RSS爬取器 | 意大利主流媒体RSS Feed | 复用TrendRadar的RSS模块 |

**OpenCLI适配器开发优先级**:

1. **ANSA** (意大利最大通讯社) — `opencli ansa breaking --category politica`
2. **Corriere della Sera** (发行量最大) — `opencli corriere search --section politica`
3. **La Repubblica** — `opencli repubblica search --section esteri`
4. **FAO Newsroom** — `opencli fao news --topic china`
5. **Twitter/X Italian** — `opencli twitter search --lang it --query "Cina OR China"`

### 6.2 过滤层: 规则引擎 + AI兴趣筛选

**从TrendRadar借鉴**:
- 两阶段过滤: 关键词规则预筛 → AI兴趣分类
- 关键词配置YAML格式（参考TrendRadar的config/config.yaml）
- 意大利特化关键词集: 涉华实体名、在意中企名、国际组织名、意大利政要名

**News Sentry增强**:
- 增加涉华相关性规则（这是TrendRadar和BettaFish都不具备的）
- 增加来源可信度评分（参考NewsEvent schema的source_credibility字段）
- 增加跨源去重（同一事件在ANSA+Corriere+Repubblica的报道合并）

### 6.3 研判层: LLM + 规则混合（参考BettaFish ForumEngine简化版）

**不建议直接使用BettaFish的ForumEngine**（过于重型），但参考其核心思想:

| ForumEngine思想 | News Sentry简化实现 |
|----------------|-------------------|
| 多Agent辩论避免思维同质化 | 研判时使用不同LLM模型/不同prompt视角交叉验证 |
| 观点提取与合成 | 研判结果包含多维度评分（涉华相关性、新闻价值、情感、紧急程度） |
| 迭代深化 | 高价值新闻触发二次深度研判（跨源验证） |
| LLM主持人协调 | Pipeline编排器充当"主持人"角色 |

**研判prompt设计要点**:
- 意大利政治语境理解（执政党联盟、在野党立场等）
- 涉华议题敏感度判断（中意关系、在意中企、一带一路等）
- 国际组织发布权威性评估（FAO/WFP/ICJ等）
- 突发事件升级趋势预判

### 6.4 输出层: Obsidian + 多渠道推送

**从TrendRadar借鉴**:
- 9渠道通知分发器（直接复用飞书/企微/Telegram推送代码）
- 消息分片逻辑（各渠道字节限制自动分片）
- HTML报告生成（带暗色模式、标签页导航）

**News Sentry增强**:
- Obsidian vault写入（.md + YAML frontmatter，参考NewsEvent schema设计）
- 按日期/来源/主题的目录结构
- Obsidian反向链接关联同一事件的多篇报道

### 6.5 数据模型: NewsEvent渐进式丰富

TrendRadar和BettaFish的数据模型各有侧重:

- **TrendRadar**: 轻合数据模型（NewsData/RSSData），偏展示
- **BettaFish**: 关系型数据模型（话题-任务-平台-内容-评论），偏分析

**News Sentry的NewsEvent schema**（已在newsevent-schema.md中设计）综合了两者的优点:
- 渐进式丰富（TrendRadar的管道式处理思想）
- 多维度评分（BettaFish的多Agent分析思想）
- 涉华相关性维度（两者都不具备，News Sentry独有）
- YAML frontmatter + Markdown双格式输出（Obsidian友好）

## 8. 风险与注意事项

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| TrendRadar强依赖NewsNow API | 国际源无法使用其热榜抓取 | 优先使用RSS + OpenCLI + World News API |
| OpenCLI需Chrome始终运行 | 服务器部署时需headless Chrome | Docker中运行Chrome + VNC/noVNC |
| BettaFish微调模型仅支持中文 | 意大利语情感分析需重新训练 | 先用LLM情感分析，后续考虑微调 |
| 社媒平台API rate limit | Twitter/X等平台有请求限制 | 合理设置请求间隔，多账号轮换 |
| 意大利新闻网站改版 | OpenCLI适配器失效 | opencli-autofix自动修复 + 监控告警 |
| LLM成本控制 | 研判环节LLM调用多 | PipelineContext.llm_budget_remaining预算控制 + 规则引擎降级 |
